Section 01

背景 — KYOEI COMPASS 2.0 という業界標準

KYOEI COMPASS 2.0 は、協栄産業が1982年から開発を続け、2,000社以上の建設会社に導入されている建築見積書作成ソフトです。 現在、当社の「見積クラウド」では案件管理・データ蓄積は実現できていますが、COMPASSが持つ「見積書そのものを作成・編集する」実務機能が不足しています。

機能 COMPASSの実装
多階層見積構造 最大16階層・明細無制限。複合ビル・大規模プロジェクトにも対応
Excel互換 Excel ↔ COMPASS の双方向コピペ。計算式付きデータ出力。1段・2段明細対応
単価マスター 地域・年度別に複数管理。市販単価データの取込。他物件からの同一明細検索で単価を自動セット
計算パターン 各明細行ごとにNET/提出の差額・利益率を確認しながら編集。計算式の自由登録
見積比較 最大10社を比較。業者単価・経費込単価・ネット単価・査定掛率単価での比較。経費振り分け選択
見積依頼（下見積） 工事科目を選択してExcel見積依頼書を出力。業者からのExcel回答を取り込んで査定。二次明細・法定福利費にも対応
自動値入 採用業者の決定後、採用単価を見積明細書に反映。別階層の同名称明細の一括値入
データ連携 BCSフォーマット、CSV、テキスト形式での出力
Section 02

COMPASSの限界 — AI で超えられるポイント

COMPASSはルールベースの自動化の限界に達しています。AIを活用することで、COMPASSでは不可能だった知的支援が実現できます。

🔍

類似案件の検索が「完全一致」止まり

名称・規格・単位の文字列一致でしか検索できない。「RC造10階建オフィス」と「SRC造12階建事務所ビル」が類似案件であることを認識できない。

💰

単価の「妥当性」を判断できない

過去単価の自動セットは可能だが、それが現在の相場と比べて妥当かどうかの判断は人間任せ。統計的な分析や異常値検出の仕組みがない。

📝

見積項目の「漏れ」に気づけない

項目生成は手動。建物種別に応じた「漏れがちな付帯工事」の提案や、類似案件との項目比較による過不足チェックができない。

🔄

業者回答の「表記ゆれ」を吸収できない

「塩ビ管VP100」「硬質塩化ビニル管100A」のような業者ごとの表記ゆれを統一するのは人手頼み。自動マッチングは不可能。

Section 03

プロダクトコンセプト

既存の見積クラウドDBを「唯一の正」として活用しながら、KYOEI COMPASSが持つ実務機能をmacOSデスクトップアプリとして実装し、AIエージェントで拡張します。

📊
COMPASS同等の見積エディタ

AG Gridベースの多階層スプレッドシートUIで、計算パターン・2段単価・利益率確認を再現。⌘+C/V によるExcel互換コピペにも対応。

🤖
AIエージェント支援

COMPASSではルールベースでしかできなかった検索・値入・チェックを、LLMが意味理解で実行。表記ゆれも自動で吸収。

☁️
見積クラウドDBとシームレス連携

案件・業者・見積データは既存の見積クラウドDBに保存。デスクトップアプリはAPI経由でリアルタイム同期し、チーム全体で一元活用。

📁
フォルダ指定で過去見積を自動取込

社内の共有フォルダを指定するだけで、Excelファイルを自動読み込み・蓄積。使えば使うほどAIが社内の見積ノウハウに詳しくなる。

🏢
元請プロファイル自動適用

ログイン時に元請プロファイル（書式・階層・計算ルール・AIチェック観点）が自動切替。「うちのやり方」に完全対応。

🔒
見積データの安全性

AI推論に必要な単価データはエンタープライズ契約のClaude API（学習利用なし）を通じて送信。原価情報はローカルに保持。

Section 05

AIで超える — 5つの差別化機能

COMPASSのルールベース自動化では実現できなかった「知的支援」を、Claude SDK の Tool Use 機能でエージェントとして実装します。

🔍

類似案件の意味検索

建物種別・規模・工種をキーに、pgvector（ベクトル検索）と条件検索のハイブリッドで類似案件を発見。文字列一致に頼らない「意味的な」類似度検索を実現。

💰

単価の統計的妥当性チェック

過去実績の統計値（平均・中央値・分布）に基づき、各項目の推奨単価を提案。相場からの乖離度を可視化し、異常値をハイライト。

📝

見積項目の自動生成・漏れ検出

建物種別と工事概要から必要な見積項目を自動生成。類似案件との比較で「漏れがちな付帯工事」も含めて提案。

🔄

表記ゆれ吸収の自動値入

採用業者の見積書から品目・単価・数量を読み取り、内訳書フォーマットに自動マッピング。業者ごとの表記ゆれをAIが吸収し、転記ミスを防止。

✅

見積書の総合レビュー

完成した見積書に対し、単価の異常値検出・項目の過不足・合計金額の妥当性を総合的に自動レビュー。人間のチェック前にAIが一次確認。

Section 09

macOS 配布・署名・CI/CD

Apple の審査プロセスに対応した署名・公証パイプラインを整備し、社内 Mac への安定配布を実現します。

🔏

Apple Developer ID 署名

Apple Developer Program（年額 $99）に加入し、Developer ID 証明書でアプリに署名。tauri.conf.json の macOS.signingIdentity に設定することで、ビルド時に自動署名される。

📋

Apple Notary Service 公証

署名済みアプリを Apple に送付し、マルウェアスキャンをパスした「公証済みチケット」を取得。macOS Gatekeeper が警告なく起動を許可する必須ステップ。

📦

.dmg 配布パッケージ

tauri build 実行で .app バンドルと .dmg を自動生成。DMG 内に Applications フォルダへのエイリアスを配置する標準レイアウトを採用。

🔁

自動アップデート

Tauri Updater プラグインで差分アップデートを配信。署名済みアップデートバイナリをサーバーに配置するだけで、次回起動時に自動で最新版に更新される。

GitHub Actions — CI/CD パイプライン 

# .github/workflows/release.yml（概要）
jobs:
  build-mac:
    runs-on: macos-latest       # Apple Silicon runner
    steps:
      - uses: actions/checkout@v4
      - name: Build & Sign Tauri App
        uses: tauri-apps/tauri-action@v0
        with:
          args: --target aarch64-apple-darwin
        env:
          APPLE_CERTIFICATE:          ${{ secrets.APPLE_CERTIFICATE }}
          APPLE_CERTIFICATE_PASSWORD: ${{ secrets.APPLE_CERT_PW }}
          APPLE_ID:                    ${{ secrets.APPLE_ID }}
          APPLE_TEAM_ID:              ${{ secrets.APPLE_TEAM_ID }}
          APPLE_PASSWORD:             ${{ secrets.APPLE_APP_PASSWORD }}
Developer ID 証明書（.p12）は Base64 エンコードして GitHub Secrets に登録。CI 実行時に自動的に Keychain に読み込まれ、公証まで一気通貫で完了します。

Section 12

リスクと対策

⚠️ 想定リスク

🍎 WKWebView（Safari エンジン）と AG Grid の CSS 互換差異
🔏 Apple 審査・公証ポリシーの変更による配布ブロック
🔐 Keychain / Entitlements の設定ミスによる起動失敗
📁 macOS Sandbox 制約でフォルダ監視が制限される可能性
🦀 Rust の学習コストによる開発遅延
💸 Claude API 利用コストの増大

✅ 対策

🛡️ Phase 1 で WKWebView / AG Grid の macOS Safari 動作確認を早期実施。CSS prefix を追加対応
🛡️ App Store 外配布（Developer ID）を基本とし、公証（Notarization）で Gatekeeper 通過を確保
🛡️ Tauri の entitlements テンプレートを使用し、必要な権限を適切に設定
🛡️ Tauri の dialog プラグイン経由でユーザーがフォルダを明示的に選択する UX で Sandbox 制約を回避
🛡️ フロントエンドは React/TS で既存知見を活用。Rust は段階的に習得
🛡️ Prompt Caching の活用とローカルキャッシュで API 呼び出しを最適化
