Section 01

背景と課題

建設業界における見積書作成は、過去案件のExcelファイルを参照しながら手作業で単価を調整する属人的な業務であり、担当者の経験と勘に大きく依存しています。

業界が直面している主要課題は以下の通りです。

⏱

見積作成に時間がかかる

過去のExcelファイルから類似案件を手動で検索し、単価を比較・調整する作業に膨大な時間を要している。

👤

属人化・ナレッジの散逸

ベテランの暗黙知がExcelファイルに分散しており、体系的な知識として蓄積・共有されていない。

⚠

単価の妥当性検証が困難

提示単価が市場相場や過去実績と乖離していないか、客観的に検証する仕組みがない。

Section 02

プロダクトコンセプト

本アプリは「AIが隣にいる見積作成環境」をコンセプトとし、Excelライクな操作感を維持しながら、LLMエージェントがリアルタイムで作成を支援するデスクトップアプリケーションです。

📊

Excel風見積エディタ

AG Gridベースのスプレッドシート UIで、既存のExcelワークフローを損なわずに編集・入力が可能。

🤖

AIエージェント支援

過去見積の類似検索、単価の自動提案、項目の自動生成、妥当性チェック、採用見積からの自動値入れをLLMが実行。

🔒

エンタープライズAPI + 個人情報保護

AI推論にはデータが学習に利用されないエンタープライズ契約API（Claude API）を使用。施主名・担当者名などの個人情報はAPI送信前に自動マスキング。

📁

データの正はクラウドに一元化

ローカルのExcelは「取込元」、見積クラウドDBが「唯一の正（マスター）」。新規作成データはすべてクラウドDBに保存し、チーム全体で一元活用。

Section 03

デスクトップアプリだからできる3つの強み

クラウドやWebアプリでは実現できない、デスクトップアプリならではの価値を3つの観点で整理しました。

効率化

フォルダを指定するだけで、過去の見積書をAIが全部覚えてくれる

いちいちファイルをアップロードする手間はありません。アプリ上で案件を作成し、社内の共有フォルダを指定しておけば、中に入っている見積書をAIが自動的に読み込み、「過去にどんな単価で出したか」「似た案件がないか」を即座に答えてくれます。

過去のExcelは「取込元」として読み取られ、データは見積クラウドDBに蓄積されます。新しい見積書がフォルダに追加されたら、それも自動で取り込みます。使えば使うほど、AIが社内の見積ノウハウに詳しくなっていく仕組みです。

見積データの安全性を確保した設計

見積書には原価や利益率など、取り扱いに注意が必要な情報が含まれています。このアプリでは、AI分析に必要な単価データはエンタープライズ契約のAPI（Claude API）を通じて送信しますが、APIプロバイダーの学習データとして利用されることはありません。

過去の見積データは自社管理の見積クラウドDBに蓄積され、ローカルのExcelファイルは自社PC内に保持されます。

Section 06

機能配置モデル — Claude Code型ハイブリッド

各機能をローカルとクラウドAPIのどちらで処理するかは、アプリの使い勝手とセキュリティを左右する重要な設計判断です。Claude Codeの「ファイル操作はローカル、知性と記憶はクラウド」という設計思想を参考に、最適なモデルを採用します。

設計原則
ローカルExcelは「入口」、見積クラウドDBが「唯一の正」、賢い判断はAI
過去のExcelファイルはローカルで読み取り・解析する「取込元」です。取り込んだデータと新規作成した見積データは、すべて見積クラウドDBに保存され、これがチーム共通のマスターデータとなります。AI推論にはエンタープライズ契約のAPI（Claude API、学習利用なし）を使用して送信します。
機能 ローカル処理 API 見積クラウドDB（正） AI推論（Claude API）
案件作成 入力UI・バリデーション → 案件の永続化 —
見積書取込 Excel解析・データ抽出 → 見積データ・単価の蓄積・ベクトル化 項目の自動分類
見積依頼 依頼書の作成UI → 依頼先・ステータス管理 依頼項目の自動生成
見積比較 比較計算・差分表示 ↔ 比較結果の保存・過去単価の参照 異常値検出・コメント生成
業者選定 選定UI・スコア表示 ↔ 業者・評価履歴・過去単価の比較 総合評価・推薦理由の生成
自動値入れ 採用見積→内訳書への転記UI → 完成した内訳書の保存 品目・単価・数量の自動マッピング

※ ローカルExcelは「取込元」としてのみ使用。取込後のデータは見積クラウドDBが唯一の正（Single Source of Truth）となります。
※ AI推論列の機能は、匿名化された単価・金額データをClaude API（エンタープライズ契約・学習利用なし）に送信して実行します。
※ 類似案件検索には、見積データをEmbeddingモデルでベクトル化し、クラウドDB上のpgvector（PostgreSQL拡張）で意味的な類似度検索を行います。

Section 07

AI支援機能の詳細

Claude SDKのTool Use機能により、以下のAI支援をエージェントとして実装します。

🔍

類似案件 自動検索

建物種別・規模・工種をキーに、過去見積DBから類似案件を自動検索。見積クラウドDB上のpgvector（ベクトル検索）と条件検索のハイブリッドで精度を確保。

💰

単価 自動提案

過去実績の統計値（平均・中央値・分布）に基づき、各項目の推奨単価を提案。相場からの乖離度も表示。

📝

項目の自動生成

建物種別と工事概要から、必要な見積項目を自動で生成。漏れがちな付帯工事も含めて提案。

✅

妥当性チェック

完成した見積書に対し、単価の異常値検出、項目の過不足、合計金額の妥当性を自動レビュー。

📥

自動値入れ

採用業者の見積書から品目・単価・数量を読み取り、内訳書のフォーマットに自動マッピング。項目名の表記ゆれもAIが吸収し、転記ミスを防止。

Section 10

リスクと対策

⚠ 想定リスク

  • Rustの学習コストによる開発遅延
  • LLM APIのレスポンス遅延がUXを損なう可能性
  • 過去見積データの品質・フォーマットのばらつき
  • AI推論のため単価データが外部APIに送信される点
  • API利用コストの増大

✅ 対策

  • フロントエンドはReact/TSで既存知見を活用、Rustは段階的に習得
  • ストリーミング応答 + UIの楽観的更新で体感速度を向上
  • インポート時のバリデーションと正規化パイプラインを構築
  • 学習利用なしのエンタープライズ契約API（Claude API）を使用
  • Prompt Cachingの活用とローカルキャッシュで API呼出しを最適化
