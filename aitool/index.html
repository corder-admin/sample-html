Section 01

背景 — KYOEI COMPASS 2.0 という業界標準

KYOEI COMPASS 2.0 は、協栄産業が1982年から開発を続け、2,000社以上の建設会社に導入されている建築見積書作成ソフトです。建設業の見積実務に必要な機能を網羅しており、事実上の業界標準として広く利用されています。

現在、当社は「見積クラウド」Webプラットフォームで案件管理やデータ蓄積を行っていますが、COMPASSが持つ「見積書そのものを作成・編集する」実務機能が不足しています。

COMPASS 2.0 の核心機能
COMPASSが2,000社に選ばれている理由は、見積書作成の実務に必要な機能を「Excel感覚の操作性」で提供している点にあります。以下がその核心機能です。

Section 02

COMPASSの限界 — AIで超えられるポイント

COMPASSは見積書作成ソフトとして高い完成度を持つ一方、2,000社が日々直面している課題は「ルールベースの自動化の限界」に集約されます。AIを活用することで、COMPASSでは不可能だった知的支援が実現できます。

🔍

類似案件の検索が「完全一致」止まり

COMPASSは名称・規格・単位の文字列一致でしか検索できない。「RC造10階建オフィス」と「SRC造12階建事務所ビル」が類似案件であることを認識できない。

💰

単価の「妥当性」を判断できない

過去単価の自動セットは可能だが、それが現在の相場と比べて妥当かどうかの判断は人間任せ。統計的な分析や異常値検出の仕組みがない。

📝

見積項目の「漏れ」に気づけない

項目生成は手動。建物種別に応じた「漏れがちな付帯工事」の提案や、類似案件との項目比較による過不足チェックができない。

🔄

業者回答の「表記ゆれ」を吸収できない

業者ごとに異なる品目名・規格表記を統一するのは人手頼み。「塩ビ管VP100」「硬質塩化ビニル管100A」のような表記ゆれの自動マッチングは不可能。

Section 05

AIで超える — 5つの差別化機能

COMPASSのルールベース自動化では実現できなかった「知的支援」を、Claude SDKのTool Use機能でエージェントとして実装します。

🔍

類似案件の意味検索

建物種別・規模・工種をキーに、見積クラウドDB上のpgvector（ベクトル検索）と条件検索のハイブリッドで類似案件を発見。文字列一致に頼らない「意味的な」類似度検索を実現。

💰

単価の統計的妥当性チェック

過去実績の統計値（平均・中央値・分布）に基づき、各項目の推奨単価を提案。相場からの乖離度を可視化し、異常値をハイライト。

📝

見積項目の自動生成・漏れ検出

建物種別と工事概要から必要な見積項目を自動生成。類似案件との比較で「漏れがちな付帯工事」も含めて提案。

🔄

表記ゆれ吸収の自動値入

採用業者の見積書から品目・単価・数量を読み取り、内訳書フォーマットに自動マッピング。業者ごとの品目名・規格の表記ゆれもAIが吸収し、転記ミスを防止。

✅

見積書の総合レビュー

完成した見積書に対し、単価の異常値検出・項目の過不足・合計金額の妥当性を総合的に自動レビュー。人間のチェック前にAIが一次確認。

Section 09

機能配置モデル — COMPASS同等機能 ＋ AI拡張の2層構造

各機能をローカルとクラウドAPIのどちらで処理するかは、アプリの使い勝手とセキュリティを左右する重要な設計判断です。Claude Codeの「ファイル操作はローカル、知性と記憶はクラウド」という設計思想を参考に、最適なモデルを採用します。

設計原則
ローカルExcelは「入口」、見積クラウドDBが「唯一の正」、賢い判断はAI
過去のExcelファイルはローカルで読み取り・解析する「取込元」です。取り込んだデータと新規作成した見積データは、すべて見積クラウドDBに保存され、これがチーム共通のマスターデータとなります。AI推論にはエンタープライズ契約のAPI（Claude API、学習利用なし）を使用して送信します。

Section 10

元請カスタマイズ対応 — 元請ごとの「うちのやり方」に応える

大手の元請会社には、見積書の書式や項目の並べ方、金額の計算方法に独自のルールがあります。同じ工事でも、A社向けとB社向けでは見積書の見た目も中身の構成も異なるのが実態です。

本アプリでは、元請ごとの独自ルールを「元請プロファイル」としてあらかじめ登録しておきます。ユーザーのアカウントには所属する元請の情報が紐づいているため、ログイン（またはAPIキー認証）するだけで、書式・構成・計算方法・AIのチェック観点がすべて自動的にその元請仕様に切り替わります。

考え方
ログインするだけで、アプリ全体が「その元請仕様」に切り替わる
ユーザーのアカウントに元請情報が登録されているため、現場で「A社向けはこの書式」「B社向けはこの計算方法」と毎回手動で切り替える必要がありません。ログインした瞬間に、その人が所属する元請のルールが自動で適用されます。新しい元請が増えても、ルールを登録してユーザーに紐づけるだけで対応できます。
元請ごとに異なるもの 具体例 本アプリでの対応方法
見積書の書式 表紙（鑑）のフォーマット、列の順番や見出し、提出用見積書の縦横、社内管理表の書式など 元請ごとのExcelテンプレートを登録。出力時に自動で差し替え
見積書の構成（階層の並べ方） A社は「工種→部位→細目」の3階層。B社は「棟別→工種→部位→細目」の4階層。必須の科目コードも元請ごとに異なる 元請ごとの「見積書の骨格（ひな型）」を登録。新規作成時に自動展開
金額の計算方法 経費の按分方法、法定福利費の扱い（別行 or 含む）、端数処理（切捨 or 四捨五入）、消費税の丸め方、査定掛率の初期値 元請ごとの計算ルールを登録。見積編集時に自動で適用
AIがチェックすべき観点 「この元請は経費込みの単価で比較する」「二次明細が必須」「仮設工事の漏れを重点チェック」など 元請ごとのチェック観点を登録。AIが自動的にその基準でチェック
🏢

元請のルールを一箇所にまとめて管理

見積書の書式、構成のひな型、計算ルール、AIのチェック観点を元請ごとにまとめて見積クラウドに登録。バラバラに管理する必要がなくなります。

🔑

ログインするだけで自動適用

ユーザーのアカウントに元請情報が紐づいているため、ログインまたはAPIキーで認証するだけで、見積書の書式・構成・計算方法・AIのチェック基準がすべて自動で切り替わります。

AIも元請のルールを理解

「この元請では仮設工事を必ず含める」といったルールをAIが自動的に読み取り、その基準に沿って見積書のチェックや提案を行います。

ER構造（テーブル間の関係）

「元請プロファイル」は1つの万能テーブルではなく、元請マスター（親）に4つの子テーブルがぶら下がる構造です。ユーザーテーブルが元請マスターへの外部キー（prime_contractor_id）を持つため、ログイン時に自動的に元請プロファイル全体に到達できます。案件テーブルは元請IDを直接持たず、作成者（ユーザー）経由で元請を解決します。

ユーザー ──FK──▶ 元請マスター // prime_contractor_id（ユーザーに元請が紐づく）
ユーザー ◀──1:N── 案件 // created_by（作成者）
案件 ◀──1:N── 見積書 // 1案件に複数版（初回・VE・追加…）

元請マスター ◀──1:N── 帳票テンプレート設定 // 用途別に複数（表紙・提出用・社内管理表…）
元請マスター ◀──1:1── 階層構造テンプレート // 初期ツリーJSON + バリデーションルール
元請マスター ◀──1:1── 計算ルール設定 // パラメータJSON
元請マスター ◀──1:1── AIコンテキストルール // チェック観点テキスト配列
ユーザーテーブルのデータ例

ユーザーテーブルが元請マスターへの外部キーを持つことで、ログインまたはAPIキー認証の時点で元請プロファイル全体が確定します。アプリ起動時にこのユーザー情報を取得し、以降のすべての操作に元請ルールを自動適用します。

// ユーザーテーブル（users）
{
  "id": "user_fujisawa",
  "name": "藤澤勇樹",
  "email": "fujisawa@example.co.jp",
  "prime_contractor_id": "pc_shimizu", // → 元請マスターへのFK
  "api_key": "sk-xxxxxxxxxxxxxxxx", // APIキー認証用
  "role": "estimator" // 権限（estimator / manager / admin）
}
案件テーブルのデータ例

案件テーブルは物件の基本情報を持ちます。元請IDは直接持たず、作成者のユーザー情報経由で元請プロファイルを解決します。これにより、同じ元請に所属する全ユーザーが同じプロファイルで案件を作成できます。

// 案件テーブル（projects）
{
  "id": "proj_20260301_001",
  "name": "○○ビル新築工事",
  "created_by": "user_fujisawa", // → ユーザーへのFK（ユーザー経由で元請を解決）
  "building_type": "RC造",
  "floors": 10,
  "total_floor_area_m2": 8500.0,
  "location": "東京都中央区",
  "status": "見積作成中",
  "created_at": "2026-03-01T09:00:00Z"
}
見積書テーブルのデータ例

1つの案件に対して複数の見積書が紐づきます（初回提出・VE案・追加工事など）。見積書も案件も元請IDを直接持たず、ログインユーザー経由で元請プロファイルを参照します。tree_data には階層構造テンプレートから展開された実際の見積ツリーが格納されます。

// 見積書テーブル（estimates）
{
  "id": "est_20260301_001",
  "project_id": "proj_20260301_001", // → 案件へのFK
  "version": 1, // 第1版（VE・追加で版が増える）
  "title": "○○ビル新築工事 見積書（初回提出）",
  "total_amount": 185000000, // 合計金額（自動計算）
  "status": "作成中", // 作成中 / レビュー中 / 提出済
  "tree_data": [ // 階層構造テンプレートから展開された実データ
    {
      "label": "建築工事",
      "category_code": "A",
      "subtotal": 120000000,
      "children": [
        {
          "label": "躯体工事",
          "category_code": "A-01",
          "subtotal": 75000000,
          "children": [
            {
              "label": "コンクリート工事",
              "category_code": "A-01-01",
              "items": [ // 末端の明細行
                {
                  "name": "普通コンクリート打設",
                  "spec": "21-18-20 スランプ18cm",
                  "unit": "m3",
                  "quantity": 1200.0,
                  "unit_price": 15500, // 提出単価
                  "net_unit_price": 13200, // NET単価（原価）
                  "amount": 18600000, // = quantity × unit_price
                  "supplier_id": "sup_abc_concrete", // 採用業者
                  "memo": ""
                },
                // ... 他の明細行
              ]
            }
          ]
        }
      ]
    }
  ],
  "created_at": "2026-03-01T09:30:00Z",
  "updated_at": "2026-03-05T14:20:00Z"
}

tree_data 内の階層構造は、見積書の新規作成時にログインユーザーの元請プロファイルから階層構造テンプレートをコピーしたものです。テンプレートの locked 属性はコピー時にバリデーションルールとして適用され、見積書データ自体には持ちません（表示時に元請テンプレートと突合してロック状態を判定します）。末端の items 配列が実際の明細行で、名称・規格・単位・数量・単価・金額を保持します。net_unit_price（NET単価）と unit_price（提出単価）の2本立てにより、COMPASSと同様に利益率をリアルタイム計算できます。

① 帳票テンプレート設定

元請ごとに「見積書表紙のExcelテンプレートファイル」「提出用見積書の列定義（どの列にどのデータを出すか）」「社内管理表のレイアウト定義」を保持します。テンプレートファイルはS3等にバイナリ保存し、DBにはパスと列マッピング定義（JSON）を格納します。用途別に複数レコードを持つ 1:N の関係です。

Excel出力時、Rust側のエクスポート処理はこの定義を読み、テンプレートファイルの該当セルにデータを流し込みます（テンプレート差込方式）。

② 階層構造テンプレート

「見積書のツリー構造の雛型」をJSON形式で保持します。例：A社は [建築工事 > 躯体工事 > コンクリート工事] の3階層、B社は [1号棟 > 建築工事 > 躯体工事 > コンクリート工事] の4階層。新規見積作成時に、このJSONからツリーのスケルトンをAG Gridに自動展開します。

各ノードには locked: true/false 属性を持たせ、ロックされた階層（変更不可）と自由階層（追加・削除可能）の境界を制御します。「この元請では科目コード必須」「この階層名は変更不可」といったバリデーションルールもここに含めます。

// 階層構造テンプレートの例（A社：工種→部位→細目 の3階層）
{
  "name": "A社 標準テンプレート",
  "max_depth": 3,
  "require_category_code": true,
  "tree": [
    {
      "label": "建築工事",
      "category_code": "A",
      "locked": true, // この階層名は変更不可
      "children": [
        {
          "label": "躯体工事",
          "category_code": "A-01",
          "locked": true,
          "children": [
            { "label": "コンクリート工事", "category_code": "A-01-01", "locked": false },
            { "label": "型枠工事", "category_code": "A-01-02", "locked": false },
            { "label": "鉄筋工事", "category_code": "A-01-03", "locked": false }
          ]
        },
        {
          "label": "仕上工事",
          "category_code": "A-02",
          "locked": true,
          "children": [] // 細目はユーザーが自由に追加
        }
      ]
    },
    {
      "label": "設備工事",
      "category_code": "B",
      "locked": true,
      "children": [] // 部位・細目はユーザーが自由に追加
    }
  ]
}

locked: true のノードは名称の変更・削除ができず、元請が定めた科目体系を保護します。locked: false のノードや children: [] の配下はユーザーが自由に項目を追加・編集できます。この構造により「元請が決めた骨格は崩さず、中身は現場で柔軟に埋める」という実務の要件を表現できます。

③ 計算ルール設定

計算ロジック自体は共通コードで、分岐に使うパラメータだけが元請ごとに異なります。JSON形式でKey-Value的に保持します。

// 計算ルール設定の例
{
  "rounding": "floor", // 端数処理：切捨
  "tax_rounding": "floor", // 消費税丸め：円未満切捨
  "legal_welfare": "separate_row", // 法定福利費：別行表示
  "expense_allocation": "proportional", // 経費按分：工事費比例
  "default_assessment_rate": 0.95 // デフォルト査定掛率
}

Rustの計算エンジンがこのJSONを受け取り、パラメータに応じた計算を実行します。新しい計算ルールが必要になった場合は、JSONにキーを追加し、計算エンジンに対応する分岐を追加します。

④ AIコンテキストルール

自然言語で書かれたチェック観点のテキスト配列です。Claude SDKのTool呼出時に、このテキストをシステムプロンプトに注入するだけなので、AI側のコード変更は不要です。

// AIコンテキストルールの例
[
  "この元請では経費込単価で業者比較を行うこと",
  "二次明細が必須。二次明細がない場合は警告すること",
  "仮設工事・安全対策費の項目漏れを重点的にチェック",
  "法定福利費は別行として表示する前提でチェック"
]
データの流れ（ログイン〜見積作成〜出力）

アプリ起動時にログインまたはAPIキーでユーザーを認証し、そのユーザーに紐づく元請プロファイルを自動取得します。以降のすべての操作にこのプロファイルが適用されます。

// 1. ログインまたはAPIキーでユーザーを認証
current_user = authenticate(login or api_key)
prime_contractor_id = current_user.prime_contractor_id

// 2. 元請IDで4テーブルからそれぞれ設定を取得（起動時に1回）
階層構造テンプレート → AG Gridの初期ツリーに適用
計算ルール設定 → Rust計算エンジンのパラメータにセット
AIコンテキストルール → Claude SDK呼出時のシステムプロンプトに注入

// 3. 見積書を新規作成する場合
階層構造テンプレート.tree → 見積書.tree_data にコピー（スケルトン展開）
→ ユーザーが明細行（items）を追加・編集していく

// 4. Excel出力時に取得
帳票テンプレート設定 → 出力用途に応じたテンプレートを選択
見積書.tree_data → テンプレートの該当セルにデータを流し込み

Section 13

リスクと対策

⚠ 想定リスク

  • Rustの学習コストによる開発遅延
  • COMPASS機能パリティの達成が想定以上に複雑
  • LLM APIのレスポンス遅延がUXを損なう可能性
  • 過去見積データの品質・フォーマットのばらつき
  • AI推論のため単価データが外部APIに送信される点
  • API利用コストの増大

✅ 対策

  • フロントエンドはReact/TSで既存知見を活用、Rustは段階的に習得
  • COMPASSの「よく使われる基本機能」から優先実装し、段階的にパリティ達成
  • ストリーミング応答 + UIの楽観的更新で体感速度を向上
  • インポート時のバリデーションと正規化パイプラインを構築
  • 学習利用なしのエンタープライズ契約API（Claude API）を使用
  • Prompt Cachingの活用とローカルキャッシュで API呼出しを最適化
