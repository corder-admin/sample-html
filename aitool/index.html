Section 01

背景 — KYOEI COMPASS 2.0 という業界標準

KYOEI COMPASS 2.0 は、協栄産業が1982年から開発を続け、2,000社以上の建設会社に導入されている建築見積書作成ソフトです。建設業の見積実務に必要な機能を網羅しており、事実上の業界標準として広く利用されています。

現在、当社は「見積クラウド」Webプラットフォームで案件管理やデータ蓄積を行っていますが、COMPASSが持つ「見積書そのものを作成・編集する」実務機能が不足しています。

COMPASS 2.0 の核心機能
COMPASSが2,000社に選ばれている理由は、見積書作成の実務に必要な機能を「Excel感覚の操作性」で提供している点にあります。以下がその核心機能です。

Section 02

COMPASSの限界 — AIで超えられるポイント

COMPASSは見積書作成ソフトとして高い完成度を持つ一方、2,000社が日々直面している課題は「ルールベースの自動化の限界」に集約されます。AIを活用することで、COMPASSでは不可能だった知的支援が実現できます。

🔍

類似案件の検索が「完全一致」止まり

COMPASSは名称・規格・単位の文字列一致でしか検索できない。「RC造10階建オフィス」と「SRC造12階建事務所ビル」が類似案件であることを認識できない。

💰

単価の「妥当性」を判断できない

過去単価の自動セットは可能だが、それが現在の相場と比べて妥当かどうかの判断は人間任せ。統計的な分析や異常値検出の仕組みがない。

📝

見積項目の「漏れ」に気づけない

項目生成は手動。建物種別に応じた「漏れがちな付帯工事」の提案や、類似案件との項目比較による過不足チェックができない。

🔄

業者回答の「表記ゆれ」を吸収できない

業者ごとに異なる品目名・規格表記を統一するのは人手頼み。「塩ビ管VP100」「硬質塩化ビニル管100A」のような表記ゆれの自動マッチングは不可能。

Section 05

AIで超える — 5つの差別化機能

COMPASSのルールベース自動化では実現できなかった「知的支援」を、Claude SDKのTool Use機能でエージェントとして実装します。

🔍

類似案件の意味検索

建物種別・規模・工種をキーに、見積クラウドDB上のpgvector（ベクトル検索）と条件検索のハイブリッドで類似案件を発見。文字列一致に頼らない「意味的な」類似度検索を実現。

💰

単価の統計的妥当性チェック

過去実績の統計値（平均・中央値・分布）に基づき、各項目の推奨単価を提案。相場からの乖離度を可視化し、異常値をハイライト。

📝

見積項目の自動生成・漏れ検出

建物種別と工事概要から必要な見積項目を自動生成。類似案件との比較で「漏れがちな付帯工事」も含めて提案。

🔄

表記ゆれ吸収の自動値入

採用業者の見積書から品目・単価・数量を読み取り、内訳書フォーマットに自動マッピング。業者ごとの品目名・規格の表記ゆれもAIが吸収し、転記ミスを防止。

✅

見積書の総合レビュー

完成した見積書に対し、単価の異常値検出・項目の過不足・合計金額の妥当性を総合的に自動レビュー。人間のチェック前にAIが一次確認。

Section 09

機能配置モデル — COMPASS同等機能 ＋ AI拡張の2層構造

各機能をローカルとクラウドAPIのどちらで処理するかは、アプリの使い勝手とセキュリティを左右する重要な設計判断です。Claude Codeの「ファイル操作はローカル、知性と記憶はクラウド」という設計思想を参考に、最適なモデルを採用します。

設計原則
ローカルExcelは「入口」、見積クラウドDBが「唯一の正」、賢い判断はAI
過去のExcelファイルはローカルで読み取り・解析する「取込元」です。取り込んだデータと新規作成した見積データは、すべて見積クラウドDBに保存され、これがチーム共通のマスターデータとなります。AI推論にはエンタープライズ契約のAPI（Claude API、学習利用なし）を使用して送信します。

Section 10

元請カスタマイズ対応 — 元請ごとの「うちのやり方」に応える

大手の元請会社には、見積書の書式や項目の並べ方、金額の計算方法に独自のルールがあります。同じ工事でも、A社向けとB社向けでは見積書の見た目も中身の構成も異なるのが実態です。

本アプリでは、元請ごとの独自ルールを「元請プロファイル」としてあらかじめ登録しておき、ログイン時にユーザーが所属する元請プロファイルを自動で取得し、適用するだけで、書式・構成・計算方法・AIのチェック観点がすべて自動的に切り替わる仕組みを採用します。

考え方
ログイン時に元請プロファイルを取得し、自動適用するだけで、アプリ全体が「その元請仕様」に切り替わる
現場で「A社向けはこの書式」「B社向けはこの計算方法」と毎回手動で切り替える必要がなくなります。元請ごとのルールを一度登録すれば、以降のすべての案件に自動で適用されます。新しい元請が増えても、ルールを登録するだけで対応できます。
元請ごとに異なるもの 具体例 本アプリでの対応方法
見積書の書式 表紙（鑑）のフォーマット、列の順番や見出し、提出用見積書の縦横、社内管理表の書式など 元請ごとのExcelテンプレートを登録。出力時に自動で差し替え
見積書の構成（階層の並べ方） A社は「工種→部位→細目」の3階層。B社は「棟別→工種→部位→細目」の4階層。必須の科目コードも元請ごとに異なる 元請ごとの「見積書の骨格（ひな型）」を登録。新規作成時に自動展開
金額の計算方法 経費の按分方法、法定福利費の扱い（別行 or 含む）、端数処理（切捨 or 四捨五入）、消費税の丸め方、査定掛率の初期値 元請ごとの計算ルールを登録。見積編集時に自動で適用
AIがチェックすべき観点 「この元請は経費込みの単価で比較する」「二次明細が必須」「仮設工事の漏れを重点チェック」など 元請ごとのチェック観点を登録。AIが自動的にその基準でチェック
🏢

元請のルールを一箇所にまとめて管理

見積書の書式、構成のひな型、計算ルール、AIのチェック観点を元請ごとにまとめて見積クラウドに登録。バラバラに管理する必要がなくなります。

🔌

ログイン時に元請プロファイルを自動で取得するだけ

案件を新規作成する時などに、ユーザーが所属する元請プロファイルで、見積書の書式・構成・計算方法・AIのチェック基準がすべて自動で切り替わります。

Tauri環境 + Excel読み書き + AG Grid多階層エディタ
Tauri 2.0プロジェクトのセットアップ、RustによるExcelパーサ/ライタの実装、AG Gridでの多階層見積書表示・編集UIを構築。計算パターン・利益率表示・Excel双方向コピペの基本CRUDを実装する。
1ヶ月
Phase 2 — 見積クラウドDB連携 + 見積比較・値入
クラウドDB連携 + 単価マスター + 見積比較 + 自動値入
既存の見積クラウドDBとのAPI連携を実装。単価マスターの地域・年度別管理、最大10社の見積比較画面（査定掛率・経費振り分け対応）、採用業者の単価反映による自動値入機能を構築する。BCS/CSV出力にも対応。
1〜2ヶ月
Phase 3 — AIエージェント統合
Claude SDK + Tool実装 + 意味検索 + 表記ゆれ吸収
Claude SDKを導入し、類似案件の意味検索・単価妥当性チェック・項目自動生成・見積書総合レビュー・表記ゆれ吸収の自動値入の各Toolを実装。pgvectorでのベクトル検索基盤も構築する。
2ヶ月
Phase 4 — 実証・改善
社内パイロット運用 + COMPASS機能パリティ検証
実業務でのパイロット運用を開始。COMPASSからの移行テストを実施し、機能パリティを検証。CSチームのフィードバックに基づきUI/UXとAI提案精度を改善する。
1ヶ月
Section 13

リスクと対策

⚠ 想定リスク

  • Rustの学習コストによる開発遅延
  • COMPASS機能パリティの達成が想定以上に複雑
  • LLM APIのレスポンス遅延がUXを損なう可能性
  • 過去見積データの品質・フォーマットのばらつき
  • AI推論のため単価データが外部APIに送信される点
  • API利用コストの増大

✅ 対策

  • フロントエンドはReact/TSで既存知見を活用、Rustは段階的に習得
  • COMPASSの「よく使われる基本機能」から優先実装し、段階的にパリティ達成
  • ストリーミング応答 + UIの楽観的更新で体感速度を向上
  • インポート時のバリデーションと正規化パイプラインを構築
  • 学習利用なしのエンタープライズ契約API（Claude API）を使用
  • Prompt Cachingの活用とローカルキャッシュで API呼出しを最適化
